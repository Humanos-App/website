§ 02 · Problem

Capital moves on assumptions,
not authorization.

Today

Agents now move money. Strategies run on their own. Execution spans treasury, exchanges, custodians, and payment rails. But the rules that govern what is allowed live in code, scattered across systems, copied from one place to another, slowly drifting out of sync, and never actually checked at the moment an action happens.

What's happening Agents act, capital moves, no one verifies.

Decisions are defined in one place. Agents act in another. Execution happens across multiple systems. Approvals — when they exist — are captured elsewhere. Nothing answers, in real time, whether a specific action is actually allowed.

decision · execution · monitoring → fragmented
What it costs Drift. Replication. Unprovable trust.

Rules duplicated per venue. Policy changes require code deploys. Auditors reconstruct trails after the fact. The firm can show what agents did — not that any given action was allowed.

no shared answer to "what is allowed right now"

Capital moves based on assumptions, not verified authorization.

§ 03 · Solution

One verification step between
decision and execution.

Humanos

Humanos is a verification layer at execution time. Every agent action is checked against a signed mandate, in real time, with one API call. When authorization is missing, Humanos collects it — instantly — and execution continues.

agent.ts · before any capital movement 
// Generate a Verifiable Presentation (VP) for the agent's action.
const { presentation } = await humanos.credentials.vp({
  credentialId: vcId,
});

// Verify before execute. One call. Deterministic answer.
const verification = await humanos.credentials.verify({
  presentation,
  action: "reallocate_capital",
  runtimeContext: {
    strategy: "delta_neutral",
    amount: 50000,
  },
});

// Execute if authorized, request approval if not.
if (verification.decision === "allow") {
  await execute(action);
} else {
  await requestApproval(verification.reason);
}
✔Authorized Execute.

Within mandate, within constraints, within validity. The agent proceeds. A Proof is generated.

✕Not authorized Recover — then continue.

Humanos requests approval instantly (SMS, API), collects KYC or identity verification if required, updates the mandate, and resumes execution once authorization is valid.

The system doesn't fail. It recovers authorization at runtime.

§ 06 · Implementation

How Numo implemented it.

01 Define intent. Users define rules — inside the Numo UI, or externally (CRM, backend, partner platforms). Sent to Humanos via API and issued as a structured mandate.
02 Agent prepares action. A strategy decides to move capital. The action is assembled with subject, amount, and context.
03 Verify before execution. Agent calls humanos.verify({...}). Constraints, validity, and revocation checked in real time.
04 Decision · recover if needed. Authorized → execution proceeds. Not authorized → Humanos requests approval or KYC in real time.
05 Resume execution · prove. Once authorization is collected, verification passes, execution continues, and a cryptographic Proof is generated.

Systems don't just check authorization — they can obtain it when needed.

§ 07 · Recovery

Handling missing authorization
in real time.

Recover

In automated systems, not every action is pre-authorized. When an agent attempts an action outside the current mandate, Humanos doesn't fail — it recovers.

01 · Detect Detect missing or invalid authorization. verify() returns a structured reason — out of scope, expired, revoked, or identity not yet verified.
02 · Request Request approval instantly. SMS, API, or email to the mandate holder. The agent waits; the system does not silently fail.
03 · Collect Collect identity verification (KYC) if required. Higher-risk actions can require step-up KYC, document verification, or biometric checks before authorization is granted.
04 · Update Update the mandate in real time. Approved scope is added to the mandate — versioned, signed, and effective immediately across every system that calls verify().
Execution resumes once authorization is valid.

Authorization is not just verified — it can be collected at the moment it's needed.

§ 09 · Network effect

Authorization becomes a shared primitive — not something rebuilt per system.

  • Mandates work across systems. Issue once. Verify anywhere — internal services, custodians, exchanges, rails.
  • Counterparties verify independently. Auditors and partners check Proofs without trusting your logs.
  • Integrations reuse the same standard. No new logic per venue. The integration is the verify() call.
  • Each new participant strengthens the standard. The more systems verify against Humanos, the more valuable a Proof becomes.
Authorization becomes a shared primitive — not something rebuilt per system.

§ 11 · Model

Issue → Verify → Collect → Prove.

I 01 · Issue Mandate. User intent → signed, structured authorization. Mandates can be created from any system — UI, backend, or external partner — and reused across all execution environments.
V 02 · Verify Check. Every agent action runs through verify(). Deterministic yes / no.
C 03 · Collect Approval. Out of bounds? Request human approval in real time — API, SMS, or email.
P 04 · Prove Receipt. Cryptographic Proof per action. Auditable forever. Verifiable by anyone.
